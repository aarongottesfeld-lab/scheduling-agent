### 'sent' value leaking into reroll update (dead code path — revised severity)

- **File:** schedule.js
- **Line:** 761
- **Description:** In the reroll route, when preserveOrgPick is false and the caller is the attendee and organizer_status === 'accepted', the update sets organizer_status: 'sent'. 'sent' violates the DB check constraint and would cause a 500. However this branch is unreachable through the UI: attendees can only single-card-reroll non-pick cards, which always sets preserveOrgPick = true. There is no "Reroll all" button in the attendee view either. The branch is only reachable via direct API calls (e.g., curl or a script).
- **Recommendation:** Replace 'sent' with 'pending' on that branch for API robustness.
- **Severity:** Medium (revised from High — dead code path through normal UI usage, latent risk for direct API callers only)

## Step 3 — Security Audit

**index.js:250** — OAuth callback redirects with userId=${sessionKey} in the URL query string. Session keys in URLs appear in browser history, server access logs, and Referer headers on cross-origin navigation.
Recommendation: Use an HTTP-only Set-Cookie header instead of a URL param.
Severity: **Critical**

**index.js:250** — supabaseId=${supabaseId} also in the redirect URL. The Supabase UUID is the user's primary key in every table. Leaking it in the URL enables cross-referencing attacks.
Recommendation: Move to cookie / encrypted payload.
Severity: **High**

**index.js:75** — userSessions is an in-memory Map. On Vercel (serverless), each function invocation may spin up a fresh instance with an empty Map → every request 401s until re-auth. The entire session model breaks on serverless deployment.
Recommendation: Persist sessions in Supabase (e.g., a sessions table) and pass a signed HTTP-only cookie.
Severity: **Critical**

**schedule.js:315** — /schedule/suggest has no per-user rate limiting. An authenticated user can call it in a tight loop, burning Anthropic API credits with no enforced ceiling (beyond the per-itinerary reroll_count). /schedule/itinerary/:id/reroll has the count cap but /suggest creates new itineraries.
Recommendation: Add a per-user daily suggestion quota stored in the DB (e.g., daily_suggest_count + last_suggest_date on profiles).
Severity: **High**
**Status: FIXED (March 12)** — Daily cap of 10/user/day enforced at schedule.js:329 using a count-only Supabase query before any profile fetches or API calls. Fails open on DB error to avoid false lockouts.

**users.js:170** — Avatar MIME type is taken from the multipart Content-Type field (client-declared), not validated against the file's actual magic bytes. A user can upload a PHP/JS payload with Content-Type: image/jpeg.
Recommendation: Run the buffer through a magic-bytes check (e.g., file-type npm package) before uploading to storage.
Severity: **High**
**Status: FIXED (March 12)** — Magic-bytes check added at users.js:164. Validates first 4 bytes against JPEG (FF D8 FF) and PNG (89 50 4E 47) signatures. Extension now derived from verified bytes, not client-declared MIME. WebP/GIF removed from allowed list.

**schedule.js:159** — contextPrompt user input is injected verbatim into the Claude prompt. The 500-char cap reduces the attack surface but prompt injection is still possible. Risk is low (authenticated users, planning context only) but Claude could be manipulated into suggesting harmful venues or leaking internal prompt structure.
Recommendation: Clarify the system/user message split; consider using the system parameter for the instruction template and user messages for inputs only.
Severity: **Medium**

**index.js:398** — Dev-user switcher is gated by !IS_PROD in JavaScript, which is correct. However, IS_PROD = process.env.NODE_ENV === 'production'. If NODE_ENV is not explicitly set in a Vercel deployment (defaults to 'production' in next.js but not plain Node.js), the gate may not engage.
Recommendation: Add an explicit NODE_ENV=production to the Vercel environment variables and add an assertion test at startup.
Severity: **Medium**

**schedule.js:356** — Claude's JSON response is parsed without schema validation. Unexpected or oversized fields from a jailbroken response (very large narrative, injected HTML/script in title) would be stored directly to the DB and rendered in the UI.
Recommendation: Validate/sanitize required fields and enforce max lengths (e.g., title ≤ 80 chars, narrative ≤ 800 chars) before writing to the DB.
Severity: **Medium**

## Step 4 — Privacy Audit

**friends.js:186** — GET /friends/:id/profile returns the target user's full profile — including bio, location, dietary_restrictions, mobility_restrictions — to any authenticated user who knows the target's UUID. There is no guard requiring the requester to be a friend. The friendshipStatus field is informational only.
Recommendation: Add a guard: if friendshipStatus !== 'accepted', return only public fields (full_name, username, avatar_url).
Severity: **High**
**Status: FIXED (March 12)** — Non-friends now receive only { id, full_name, name, username, avatar_url, friendshipStatus }. Full profile gated behind friendshipStatus === 'accepted'.

**schedule.js:325–326** — Full user profiles including dietary_restrictions and mobility_restrictions (health data) are passed to Anthropic's API. This happens without explicit user disclosure.
Recommendation: Add a privacy notice in the app. Review Anthropic's data retention policy for API inputs and ensure it meets your obligations.
Severity: **High**

**schedule.js:458** — console.log('[confirm] Received:', { itineraryId, suggestionId, isSuggestAlternative, userId: req.userId }) — logs user IDs and itinerary IDs to stdout. In production (Vercel), these go to the log storage product which may be retained and searchable.
Recommendation: Remove or reduce to a debug-only log behind if (!IS_PROD).
Severity: **Medium**

**users.js:35** — share_token is returned in every /users/me response. If this is a capability token for sharing a profile link, it shouldn't be returned on every profile fetch — only when the user explicitly requests it.
Recommendation: Move share_token to a separate, intentional endpoint (GET /users/me/share-link).
Severity: **Low**

## Step 5 — Disaster Recovery Audit

**index.js:75** — In-memory session store. Server restart = all sessions lost, all users logged out mid-session.
Recommendation: Persist sessions in Supabase before any production deployment.
Severity: **Critical**

**schedule.js:840** — Draft delete is a hard DELETE (no soft-delete / tombstone). If the organizer accidentally deletes a draft, it's unrecoverable without a DB backup.
Recommendation: Add deleted_at column + soft-delete filter, or at minimum enable Supabase point-in-time recovery.
Severity: **Medium**

**schedule.js:638** — reroll_count >= 10 permanently blocks all rerolls including appendMode. There is no reset path. Users who exhausted rerolls on an old itinerary (before appendMode was added) are permanently locked out of generating more options.
Recommendation: Either exclude appendMode from the counter, or add an admin/support tool to reset the count.
Severity: **Medium**

**schedule.js:569–583** — On lock: locked_at is written first, then calendar_event_id in a separate update (two DB round-trips, no transaction). If the server crashes or the second write fails, the plan is locked without a calendar event reference, and future calls to the confirm endpoint check !calendarEventId and retry — meaning a second calendar event could be created.
Recommendation: Wrap in a single update call by deferring the calendar creation to a background job, or store calendar_event_id in the same atomic update as locked_at.
Severity: **Low**

**index.js:78–83** — oauthStates cleanup interval runs every 5 minutes via setInterval. On serverless (Vercel), this interval never fires because each invocation is stateless. CSRF state tokens accumulate infinitely within a single warm instance. Low real-world risk since the Map resets on cold start.
Recommendation: Moot after switching to a persistent session store.
Severity: **Low**
